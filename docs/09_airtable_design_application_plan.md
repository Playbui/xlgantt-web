# Airtable Design 적용 계획

## 기준

- 루트의 `DESIGN.md`를 Airtable 계열 디자인 기준 문서로 둔다.
- 원문은 마케팅 사이트 기준이므로, XLgantts에는 "구조화된 업무 데이터 UI" 관점으로 변환해서 적용한다.
- WBS/이슈/통계/관리자 화면은 밀도와 가독성이 우선이므로 큰 히어로, 과한 여백, 랜딩식 섹션 리듬은 적용하지 않는다.

## 적용 원칙

- 기본 바탕은 흰색 캔버스와 얇은 hairline border를 사용한다.
- primary action은 파란색보다 near-black 계열을 우선한다.
- 배지는 coral, forest, cream, mint, yellow 계열을 상태/구분 의미에 맞게 제한적으로 쓴다.
- 카드 radius는 10-12px 이하로 유지한다.
- 그림자는 최소화하고, 섹션 구분은 헤더 배경색과 테두리로 처리한다.
- 업무 화면의 본문 폰트는 현재 Pretendard를 유지하되, 굵기는 과하게 올리지 않는다.

## 화면별 적용 순서

1. 프로젝트 목록
   - 프로젝트 카드, 상태 배지, 카테고리 그룹 헤더를 Airtable 톤으로 정리한다.
   - 현재 프로젝트 상황판의 과한 장식성은 줄이고 정보 밀도를 유지한다.

2. 프로젝트 진입 화면
   - WBS/이슈 선택 화면은 두 권한 모두 있는 경우에만 표시한다.
   - 선택 카드의 헤더, 아이콘, CTA를 `DESIGN.md` 기준의 차분한 카드로 정리한다.

3. 이슈 트래커
   - 목록/상세/입력/처리/공수 로그 섹션을 hairline card와 muted header 구조로 통일한다.
   - 상태/구분/우선순위 배지 색상 체계를 정리한다.
   - 입력 공간이 좁아지지 않도록 상세 패널 우선 레이아웃은 유지한다.

4. 이슈 통계
   - 요약 수치, 랭킹, 분포, 추이를 Airtable식 데이터 카드로 정리한다.
   - 프로젝트 시작일/종료일/진행 기준일/진행률 표시는 통계 카드 안에서 일관되게 보여준다.

5. 관리자
   - 사용자/조직/프로젝트 권한 테이블은 spreadsheet-database 느낌을 살린다.
   - 셀렉트, 체크박스, role chip의 간격과 색상만 정리하고 동작은 유지한다.

6. WBS 본화면
   - 간트/표 자체는 크게 손대지 않는다.
   - 헤더, 툴바, 버튼, 배지, 설정 패널의 토큰만 마지막에 맞춘다.

## 보류

- 랜딩 페이지식 96px section rhythm은 업무 화면에는 적용하지 않는다.
- pill button은 일반 업무 화면에 남발하지 않는다.
- signature coral/forest 대형 카드는 통계나 빈 상태처럼 강조가 필요한 곳에만 쓴다.
